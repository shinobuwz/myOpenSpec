## 新增需求

### 需求:TDD 红阶段写入测试结果

在红阶段运行测试并确认失败后，AI 必须立即将结果追加写入 `test-report.md`，记录：当前 task 编号、阶段标识（🔴 红）、执行时间戳、失败的测试用例名称列表及失败原因摘要。不得等到绿阶段或退出契约时再写。

#### 场景:红阶段测试运行后立即写入
- **当** AI 在红阶段运行测试命令并收到失败结果
- **那么** AI 必须在进入绿阶段前，将本次失败结果写入 `test-report.md` 对应 task 节
- **且** 记录内容包含：时间戳、失败测试用例列表、失败原因摘要

#### 场景:红阶段测试意外通过（反模式检测）
- **当** AI 在红阶段运行测试命令，测试却通过了
- **那么** AI 必须停止，在 `test-report.md` 中记录"红阶段未失败"警告
- **且** 提示用户：测试应先失败，当前测试可能已覆盖目标行为，或测试逻辑有误

### 需求:TDD 绿阶段写入测试结果

在绿阶段运行测试并确认通过后，AI 必须立即更新 `test-report.md`，在对应 task 节追加：阶段标识（🟢 绿）、执行时间戳、通过的测试用例名称列表。

#### 场景:绿阶段测试通过后立即写入
- **当** AI 在绿阶段运行测试命令并收到通过结果
- **那么** AI 必须在进入重构阶段前，将本次通过结果写入 `test-report.md` 对应 task 节
- **且** 记录内容包含：时间戳、通过测试用例列表

### 需求:TDD 重构阶段写入测试结果

在重构阶段运行测试并确认仍通过后，AI 必须立即更新 `test-report.md`，在对应 task 节追加：阶段标识（♻️ 重构）、执行时间戳、通过测试用例列表、覆盖率数字（若测试框架支持则记录，否则填 N/A）。

#### 场景:重构阶段测试通过后立即写入
- **当** AI 在重构阶段运行测试命令并收到通过结果
- **那么** AI 必须将本次结果写入 `test-report.md` 对应 task 节
- **且** 记录内容包含：时间戳、通过测试用例列表、覆盖率（支持则填数字，否则 N/A）

### 需求:test-report.md 格式规范

`test-report.md` 必须遵循固定结构：文件级标题 + 变更名，每个 task 为独立 `##` 节，节内按红→绿→重构顺序排列三个子节，每个子节包含时间戳、测试用例列表（通过/失败状态）、验收标准覆盖对应表（仅绿阶段填写）。

#### 场景:首次写入时创建文件
- **当** 当前变更目录下不存在 `test-report.md`
- **且** AI 即将写入第一条记录
- **那么** AI 必须先创建文件，写入文件级标题 `# Test Report: <change-name>`

#### 场景:后续写入时追加内容
- **当** `test-report.md` 已存在
- **且** AI 需要写入新阶段的结果
- **那么** AI 必须追加到文件末尾或对应 task 节内，禁止覆盖已有记录

#### 场景:验收标准覆盖对应表
- **当** AI 在绿阶段写入结果
- **那么** 必须在该 task 节内包含一个表格：每行对应一条验收标准，列出覆盖该标准的测试用例名称和通过状态

#### 场景:红阶段子节最小字段集
- **当** AI 在红阶段写入结果
- **那么** 红阶段子节必须包含：时间戳、失败测试用例列表、失败原因摘要
- **且** 红阶段子节中**不包含**验收标准覆盖对应表

#### 场景:重构阶段子节最小字段集
- **当** AI 在重构阶段写入结果
- **那么** 重构阶段子节必须包含：时间戳、通过测试用例列表、覆盖率数字（支持则填数字，否则 N/A）
- **且** 重构阶段子节中**不包含**验收标准覆盖对应表
