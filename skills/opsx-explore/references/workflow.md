# opsx-explore Workflow

## OpenSpec 上下文检查

开始时快速检查当前项目是否已有活动变更：

```bash
opsx changes list
```

若用户提到某个 change，读取相关产物作为上下文：

- `openspec/changes/<name>/proposal.md`
- `openspec/changes/<name>/design.md`
- `openspec/changes/<name>/specs/**/*.md`
- `openspec/changes/<name>/tasks.md`

讨论中自然引用已有产物，例如“现有 design 选择了 Redis，但现在约束看起来更适合 SQLite”。不要为了形式强行读取无关产物。

## 不同切入点

### 模糊想法

目标是打开问题空间，帮助用户看到能力谱系、约束和关键取舍。可以用 ASCII 图或对比表说明方向。

### 具体问题

先调查现实代码或文档，再把发现组织成 2-3 个最可能的方向。不要只给抽象建议。

### 实施中途卡住

读取当前 change 的产物和任务进度，定位卡点属于需求、设计、实现、验证还是范围问题。必要时建议更新 OpenSpec 产物或补一个探针任务。

### 方案比较

比较表至少包含：适用条件、优点、代价、主要风险、失效场景。可以推荐一个方案，但要说明推荐依据和不适用条件。

## 收敛阶段

触发信号：

- 用户说“差不多了”“方向明确了”“准备开始”。
- 讨论已经收敛到明确问题和方向。
- 当前信息足以决定下一步 workflow。

收敛要求：

1. 提出 2-3 个可行方案；如果只有 1 个，说明其他路线为什么不成立。
2. 每个方案包含核心思路、适用条件、优点、代价和主要风险。
3. 明确推荐方案，并说明失效条件。
4. 先确认方向，再分段确认范围、架构、数据流、错误处理、测试策略、迁移或兼容性。
5. 每段后停下来确认，不把大段模糊设计一次性要求用户批准。

## 转出规则

探索结束可能有三种结果：

- 收敛到低风险小改动或明确缺陷修复：建议 `opsx-fast`。
- 收敛到正式变更：必须先 `opsx-slice`，由 slice 定义交付边界，再进入 `opsx-plan`。
- 只澄清了问题：总结已知事实、未决问题和后续建议。

标准提示：

```text
探索已收敛，下一步按范围选择：

-> opsx-fast  （快速工作项：低风险小改动或明确缺陷修复）
-> opsx-slice （正式变更：切分判定，定义交付边界）
   slice 完成后再进入 opsx-plan
```
