# Red Green Refactor

## 红

1. 根据需求编写一个最小失败测试。
2. 运行测试，确认失败且失败原因正确。
3. 不要一次写多个测试。
4. 合法红包括断言失败或产品代码缺失。
5. 假红包括测试语法、拼写、import 路径等错误；必须先修复测试，再重跑到合法红。
6. 立即写入 `test-report.md` 的红阶段。

## 绿

1. 编写刚好通过测试的最少代码。
2. 运行测试确认通过。
3. 不提前优化。
4. 立即写入 `test-report.md` 的绿阶段，记录通过用例、变更摘要、测试变更记录和验收标准覆盖表。

`[characterization-first]` 中从旧行为断言改到目标行为是预期操作，但必须逐条记录：哪个测试、旧断言、新断言、行为变更理由。

## 重构

1. 在测试通过保护下重构。
2. 消除重复、改善命名、简化逻辑。
3. 运行测试确认未破坏功能。
4. 立即写入 `test-report.md` 的重构阶段，记录通过用例、变更摘要和覆盖率。
