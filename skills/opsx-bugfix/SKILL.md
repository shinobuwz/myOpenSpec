执行精简 bugfix 工作流。目标是在明确缺陷上少走文档流程，但不跳过根因和验证。

## 适用条件

- 问题边界相对清晰。
- 主要目标是修复缺陷，而不是扩展功能。
- 已有症状、预期行为或复现/观察线索。
- 不需要大范围需求脑暴或复杂方案比较。

## 根因铁律

没有根因，不允许修复。修复前必须能回答：

- 现象、预期和复现/观察方式是什么？
- 坏值、错误状态或异常行为从哪里产生？
- 哪个调用链、配置、状态转移或最近变更最可能导致问题？
- 当前是否只有一个明确假设？

同一问题连续 3 次修复失败时，停止叠加补丁，重新审视架构或问题定义。

## 输入 / 输出边界

**读取：**
- 用户提供的症状、预期行为、复现线索
- `.aiknowledge/codemap/` 和 `.aiknowledge/pitfalls/`
- 相关代码、日志、已有测试

**写入：**
- 相关实现代码和测试代码
- `.aiknowledge/pitfalls/`（新增或更新经验）
- `.aiknowledge/codemap/`（仅在模块边界或链路变化时）

**禁止写入：**
- 默认不创建 proposal、design、specs、tasks
- 不写 OpenSpec gates；问题扩大为需求变更时切回常规流程

## 快速流程

1. 确认 bug 描述；缺少症状、期望或线索时先问。
2. 读 codemap 辅助定位，按需读取 pitfalls。
3. 决定测试策略：`test-first`、`characterization-first` 或纯配置类 `direct`。
4. 形成单一假设，写一句根因假设：“我认为根因是 X，因为证据 Y”。
5. 用最小操作验证单一假设；假设被否定则回到调查。
6. 做最小修复并运行相关验证。
7. 按需调用 `opsx-codemap` 或 `opsx-knowledge` 沉淀知识。

## Reference 导航

- `references/workflow.md`：codemap/pitfall 读取、测试策略、修复、验证和知识沉淀细节。

## 护栏

- 不为简单 bugfix 强行创建正式 OpenSpec 产物。
- 不因为是 bugfix 就跳过测试。
- 不扩展范围；变成功能开发时暂停并切回常规流程。
- 不猜修；修复必须对应已确认根因和证据。
