测试驱动开发 Skill。强制执行红绿重构循环，确保所有产品代码都有测试覆盖。

## 铁律

**先写测试，再写代码。没有例外。** 任何产品代码的编写都必须遵循 TDD 循环。如果发现自己在没有失败测试的情况下编写产品代码，立即停止。

## 输入 / 输出边界

**读取上游产物：**
- 当前 task 的需求、验收标准、执行标签
- 命中的 `.aiknowledge/pitfalls/`
- 项目的测试框架配置
- 当前 task 涉及的实现文件和既有测试文件

**写入自身产物：**
- 测试代码和对应实现代码
- `openspec/changes/<name>/test-report.md`

**边界约束：**
- tdd 不写 `.openspec.yaml` 的 gates；其状态只在 `test-report.md` 中留档
- tdd 不要求执行 git 提交；提交是否发生由更高层流程或用户显式决定

## 启动序列

1. 确认当前任务的需求和验收标准
2. 从 task 标签读取执行方式：`[test-first]` / `[characterization-first]` / `[direct]`
3. 只有标注为 `[test-first]` 的任务强制走完整红绿重构循环；`[characterization-first]` 先固化旧行为；`[direct]` 仅用于纯样式、纯配置、纯脚手架
4. 探测项目使用的测试框架
5. 确认测试命令可以正常运行
6. 读取 `.aiknowledge/pitfalls/` 中与当前任务相关领域的 index.md；如命中相关条目，继续读取 L3 具体条目文件（`<domain>/<slug>.md`），获取详细的修复模式和根因分析，写测试时覆盖已知易错点

## 测试框架探测

| 项目标志文件 | 测试框架 |
|-------------|----------|
| CMakeLists.txt | gtest |
| pyproject.toml / setup.py | pytest |
| package.json | vitest / jest |
| build.gradle / pom.xml | JUnit |
| *.xcodeproj | XCTest |

## 红绿重构循环

在进入循环前，先判断任务模式：

- `[test-first]`：严格执行下面的红 → 绿 → 重构
- `[characterization-first]`：先写”当前行为固化测试”，确认旧行为可复现，再修改代码并补齐目标测试。**注意**：此模式下修改固化测试的断言（从旧行为→新行为）是预期操作，但仍必须在绿阶段的「测试变更记录」中逐条记录 `旧断言 → 新断言 → 行为变更理由`
- `[direct]`：不调用本 skill，直接在 implement 中执行

### 红 - 写失败测试
1. 根据需求编写一个最小的失败测试
2. 运行测试，确认它失败（且失败原因正确）
3. 不要一次写多个测试
4. **判断失败是否为合法红**：
   - **合法红**：断言失败（`AssertionError`）或产品代码缺失（`ImportError`、`ModuleNotFoundError`）→ 继续绿阶段
   - **假红**：测试代码自身的语法错误、拼写错误、import 路径错误等 → 必须先修复测试代码，重新运行直到得到合法红，并在 test-report.md 中记录修正过程
5. 立即写入 `test-report.md`：若文件不存在，先创建并写入标题 `# Test Report: <change-name>`；在对应 task 节内追加 `### 🔴 红阶段` 子节，记录时间戳、失败测试用例列表（含原始错误类型和简要归因）、假红修正记录（如有）。若红阶段测试意外通过，记录"红阶段未失败"警告并停止，提示用户检查测试逻辑。

### 绿 - 写最少代码通过测试
1. 编写刚好能让测试通过的最少代码
2. 运行测试，确认它通过
3. 不要过度设计，不要提前优化
4. 立即写入 `test-report.md`：在对应 task 节内追加 `### 🟢 绿阶段` 子节，记录时间戳、通过测试用例列表、变更摘要（修改了哪些文件、改动类型、一句话意图）、测试变更记录（如有）、验收标准覆盖对应表。
   - **变更摘要**：列出每个修改的文件及改动类型（新增/修改/删除）和一句话意图
   - **测试变更记录**：如果修改了红阶段已写测试的断言逻辑或删除了测试用例，必须逐条记录 `哪个测试 → 改了什么 → 为什么改`；如果没有修改测试，写"无测试修改"。新增测试和 setup 调整不需要记录

### 重构 - 改善代码质量
1. 在测试通过的保护下重构代码
2. 消除重复、改善命名、简化逻辑
3. 运行测试，确认重构没有破坏功能
4. 立即写入 `test-report.md`：在对应 task 节内追加 `### ♻️ 重构阶段` 子节，记录时间戳、通过测试用例列表、变更摘要（修改了哪些文件、改动类型、一句话意图）、覆盖率（测试框架支持则填数字，否则填 N/A）。

## `[manual]` 验收标准处理

验收标准中标注 `[manual]` 的项无法通过自动化测试覆盖，不阻塞 TDD 循环完成。处理规则：

1. TDD 红绿重构循环仅针对非 `[manual]` 的验收标准执行
2. `[manual]` 项在 test-report.md 对应 task 节末尾单独列出 `### ⏳ 待人工验证` 清单
3. 格式：`- [ ] [manual] <验收标准内容> — 验证方法：<方法描述>`

## 完成条件

- 所有非 `[manual]` 验收标准都有对应的测试
- 所有测试通过
- 代码已经过重构
- 测试覆盖了边缘情况
- `[manual]` 验收标准已记录到 test-report.md 的待人工验证清单

## test-report.md 格式规范

`test-report.md` 是变更级实时测试留档文件，禁止覆盖已有记录，必须追加写入。

**文件结构：**
```
# Test Report: <change-name>

## Task N: <描述>

### 🔴 红阶段
- **时间戳**：YYYY-MM-DD HH:MM
- **失败测试列表**：
  - `<test-case-name>` — `<ErrorType>`: <简要归因>（产品代码未实现 / 断言失败 / 测试代码错误）
- **假红修正**：<如有，记录修正内容：哪个测试文件 → 改了什么 → 修正后重跑结果；无则写"无">

### 🟢 绿阶段
- **时间戳**：YYYY-MM-DD HH:MM
- **通过测试列表**：
  - `<test-case-name>` ✅
- **变更摘要**：
  - `<file-path>` — <新增|修改|删除> — <一句话意图>
- **测试变更记录**：<无测试修改 | 逐条记录：`<test-name>` → 改了什么 → 为什么改>
- **验收标准覆盖对应表**：

| 验收标准 (AC) | 测试用例名称 | 通过状态 |
|--------------|------------|---------|
| ACN: ...     | `<test-name>` | ✅ |

### ♻️ 重构阶段
- **时间戳**：YYYY-MM-DD HH:MM
- **通过测试列表**：
  - `<test-case-name>` ✅
- **变更摘要**：
  - `<file-path>` — <新增|修改|删除> — <一句话意图>
- **覆盖率**：xx%（或 N/A）
```

**字段约束：**
- 红阶段子节：包含时间戳、失败测试列表（含错误类型和归因）、假红修正；**不含**验收标准覆盖对应表、变更摘要
- 绿阶段子节：包含时间戳、通过测试列表、变更摘要、测试变更记录、验收标准覆盖对应表；**不含**覆盖率
- 重构阶段子节：包含时间戳、通过测试列表、变更摘要、覆盖率；**不含**验收标准覆盖对应表、测试变更记录
- 追加语义：每次写入必须追加到文件末尾或对应 task 节内，严禁覆盖已有记录

## 退出契约

- 所有新增/修改的代码已有对应测试（`[manual]` 项除外）
- 测试全部通过
- 输出总结：新增了哪些测试、覆盖了哪些行为
- `test-report.md` 已包含所有 task 的红→绿→重构三阶段记录
- 如存在 `[manual]` 项，输出待人工验证清单及风险提示

## 反模式

以下行为严格禁止：
1. 先写产品代码再补测试 - 参见 @anti-patterns.md
2. 写了测试但跳过红色阶段（测试一开始就通过）- 参见 @anti-patterns.md
3. 一次性写大量测试而不逐个循环 - 参见 @anti-patterns.md
4. 重构时不运行测试 - 参见 @anti-patterns.md
5. 用 mock 替代真实行为测试而导致假阳性 - 参见 @anti-patterns.md

## 合理化借口防护

| 借口 | 正确做法 |
|------|----------|
| "这个函数太简单了，不需要测试" | 简单函数也需要测试，因为它可能变复杂 |
| "我先写代码理清思路，再补测试" | 用测试来理清思路，这就是 TDD 的目的 |
| "测试框架还没配好" | 先配好测试框架再写代码 |
| "这是 UI 代码，不好测试" | 分离逻辑层，逻辑层必须测试 |
| "时间不够写测试了" | 没有测试的代码不算完成 |
